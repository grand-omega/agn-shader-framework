## What worked well

- The `DType.__init__` pattern (lines 33-35) is a clean way to attach two pieces of metadata (`name_str`, `itemsize`) to each enum member while keeping each member's definition readable.
- `tensor_size` delegates unit conversion to `convert_memory`'s underlying arithmetic rather than duplicating it, then calls `tensor_size` itself from `estimate_tensor_count` — good layering that avoids duplication (`src/memory_calc.py:110`).
- Test classes (`TestConvertMemory`, `TestTensorSize`, `TestEstimateTensorCount`) mirror the three public functions cleanly and are easy to navigate.
- Every `ValueError` message is informative: includes the bad value (e.g., `src/memory_calc.py:53`, `81`, `108`).

## Issues found

No blocking issues. One minor observation:

- **`estimate_tensor_count` does not independently validate `shape`** (`src/memory_calc.py:86-111`). It relies on `tensor_size` to raise `ValueError` for empty/zero-dimension shapes, which it does correctly (via delegation at line 110). This is fine functionally, but the docstring at line 104 promises the raise — that promise is fulfilled transitively, not directly. This is acceptable given the plan spec and the tests pass, but worth noting for readability.

## Suggestions

- Consider adding a `test_estimate_invalid_shape_raises` test case in `tests/test_memory_calc.py` to explicitly exercise the shape-validation path through `estimate_tensor_count` (currently the path is covered by coverage but not exercised via a named test in that class).
- The `name_str` attribute on `DType` is defined but never used by any of the three public functions or tests. It matches the plan spec, so its absence from tests is not a defect, but a future `dtype_info()` helper or `__repr__` could make use of it.

## Lessons learned

- Layering validation through delegation (having `estimate_tensor_count` call `tensor_size` for shape validation rather than re-implementing the check) keeps the code DRY but can obscure which layer is responsible; documenting this delegation explicitly in the docstring would help future readers.
- 100% coverage on the new module was achievable without effort because the module is small, stateless, and well-decomposed — a good signal that the plan's "stateless, side-effect-free" design goal was followed.
